Validation Process

The framework validates data at each stage of the pipeline

📊

1. Profile Your Data

The DataProfiler analyzes your data to understand patterns, types, and distributions.

  • Detects column types (ID, date, numeric, categorical, text)
  • Calculates null percentages and uniqueness ratios
  • Identifies common patterns and value ranges
  • Generates a quality score for each column
→
📝

2. Generate Expectations

Based on profiling, the framework auto-generates YAML validation rules.

  • Configurable null tolerance per column
  • Uniqueness checks for ID columns
  • Type-appropriate range and format checks
  • Severity levels: critical, high, medium, low
→
✅

3. Validate at Runtime

The DataQualityValidator runs expectations against your data.

  • Uses Great Expectations engine under the hood
  • Spark-native mode for large datasets (no memory issues)
  • Returns detailed pass/fail results with diagnostics
  • Configurable actions: log, alert, or halt pipeline
→
📈

4. Act on Results

Based on validation results, the framework takes appropriate action.

  • Bronze: Log failures, continue processing
  • Silver: Send alerts, quarantine bad data
  • Gold: Halt pipeline, prevent bad data reaching reports
  • Results saved to Lakehouse for trend analysis

Validation Strategies by Dataset Size

< 100K rows Standard

Direct pandas validation

validator.validate(df)
100K - 1M rows Auto-Sample

Automatic sampling enabled

sample_large_data=True
1M - 10M rows Chunked

Process in memory-safe chunks

chunk_size=100000
> 10M rows Spark-Native

Validate directly on Spark

use_spark_native=True

Quick Start

Choose your environment and get up and running

📦
Reproducible Environment: This project uses a dedicated Conda virtual environment with all dependencies pre-configured. We recommend using this for consistent behavior.
1

Clone the Repository

Terminal 
git clone https://github.com/hs2/data-quality-library.git
cd data-quality-library
2

Activate the Conda Environment

The project includes a reproducible Conda environment with all dependencies:

Terminal 
# Create environment from the project's environment.yml
conda env create -f environment.yml

# Activate the environment
conda activate dq_framework

# Verify installation
python -c "from dq_framework import DataProfiler; print('✅ Ready!')"
⚠️
Note: Using pip install directly is not recommended as it may result in dependency conflicts. Always use the Conda environment.
3

Profile Your Data & Generate Config

Python 
from dq_framework import DataProfiler
import pandas as pd

# Load your data
df = pd.read_parquet("my_data.parquet")

# Profile and generate validation config
profiler = DataProfiler(df)
config = profiler.generate_expectations(
    validation_name="my_table_validation",
    null_tolerance=5.0
)

# Save YAML config for reuse
profiler.save_config(config, "configs/my_table.yml")
4

Run Validation

Python 
from dq_framework import DataQualityValidator

validator = DataQualityValidator("configs/my_table.yml")
results = validator.validate(df)

if results["success"]:
    print(f"✅ Validation passed: {results['success_rate']:.1f}%")
else:
    print(f"❌ Failed: {results['failed_checks']} checks failed")
☁️
Workspace Environment: In Microsoft Fabric, the library must be added to a Workspace Environment and published before notebooks can use it. This ensures consistent library versions across your workspace.
1

Create or Configure Workspace Environment

Navigate to your Fabric Workspace and set up an Environment:

  1. Open your Fabric Workspace
  2. Go to Manage → Environments
  3. Click + New Environment or select existing
  4. Name it (e.g., dq_environment)
2

Add the DQ Framework Library

Upload the library package to your environment:

  1. In the Environment settings, go to Public libraries
  2. Add required PyPI packages: 
    great-expectations==0.18.*
pyyaml>=6.0
pandas>=2.0
  3. Go to Custom libraries
  4. Upload fabric_data_quality-2.0.0-py3-none-any.whl
3

Publish the Environment

  1. Click Publish to make the environment available
  2. Wait for the environment to build (may take 2-5 minutes)
  3. Verify status shows Published
⏱️
Important: Environment changes require re-publishing. Notebooks will not see library updates until the environment is re-published.
4

Attach Environment to Notebook

  1. Open your Fabric Notebook
  2. Click the Environment dropdown (top bar)
  3. Select your dq_environment
  4. The notebook will restart with the new environment
5

Generate or Upload Configs

You can auto-generate validation configs directly in Fabric, or upload pre-made configs:

Option A: Generate in Notebook (Recommended)

Fabric Notebook 
from dq_framework import DataProfiler

# Load your table
spark_df = spark.table("lakehouse.bronze.incidents")
df = spark_df.limit(100000).toPandas()  # Sample for profiling

# Auto-generate validation config
profiler = DataProfiler(df)
config = profiler.generate_expectations(
    validation_name="bronze_incidents",
    null_tolerance=10.0
)

# Save to Lakehouse Files
profiler.save_config(config, "/lakehouse/default/Files/configs/bronze_incidents.yml")

Option B: Upload Pre-made Configs

Or manually upload YAML files to Files/configs/ in your Lakehouse.

6

Run Validation in Notebook

Fabric Notebook 
# Framework is available via the Workspace Environment
from dq_framework.fabric_connector import FabricDataQualityRunner

# Load Delta table from Lakehouse
spark_df = spark.table("lakehouse.bronze.incidents")

# Initialize runner with config from Lakehouse Files
runner = FabricDataQualityRunner("Files/configs/bronze_incidents.yml")

# Validate with Spark-native mode (recommended for large tables)
results = runner.validate_spark_dataframe(
    spark_df,
    use_spark_native=True
)

# Results auto-saved to Files/dq_results/
print(f"Validation: {'✅ Passed' if results['success'] else '❌ Failed'}")
print(f"Success rate: {results['success_rate']:.1f}%")

Local Development

Full functionality on your local machine

✓ Pandas DataFrame validation
✓ CSV, Parquet, Excel, JSON support
✓ Auto-profiling for config generation
✓ Large file auto-sampling
Python 
from dq_framework import DataProfiler, DataQualityValidator
from dq_framework.loader import DataLoader

# Load with auto-sampling for large files
df = DataLoader.load_data("large_file.parquet", sample_size=100000)

# Profile with type overrides (bypass auto-detection)
profiler = DataProfiler(
    df,
    type_overrides={"customer_id": "id"},  # Force as ID type
    strict_date_detection=True              # Reject numeric date-like IDs
)

# Generate with per-column null tolerances
config = profiler.generate_expectations(
    validation_name="customer_data",
    null_tolerances={
        "customer_id": 0.0,      # No nulls for primary key
        "optional_notes": 20.0   # 20% nulls OK for optional field
    }
)

profiler.save_config(config, "customer_validation.yml")

API Reference

Core classes and methods

class

DataProfiler

Analyzes data to automatically generate validation expectations.

DataProfiler(df, sample_size=None, type_overrides=None, strict_date_detection=False)

Parameters

df DataFrame to profile
sample_size Rows to sample (default: all)
type_overrides Dict mapping columns to types
strict_date_detection Reject numeric date-like IDs

Key Methods

  • profile() → Returns profiling statistics
  • generate_expectations(...) → Returns YAML config dict
  • save_config(config, path) → Saves to YAML file
class

DataQualityValidator

Core validation engine using Great Expectations.

DataQualityValidator(config_path=None, config_dict=None)

Parameters

config_path Path to YAML config file
config_dict Config as dictionary (alternative)

Key Methods

  • validate(df, batch_name) → Returns results dict
  • get_expectation_list() → Returns configured expectations
  • get_config_summary() → Returns config overview
class

FabricDataQualityRunner

MS Fabric-specific runner with Spark support and Lakehouse integration.

FabricDataQualityRunner(config_path, workspace_id=None, results_location="Files/dq_results")

validate_spark_dataframe()

Validate Spark DataFrame with multiple strategies

spark_df Spark DataFrame
sample_large_data Auto-sample >100K (default: True)
use_spark_native Use SparkDFDataset (no toPandas)
chunk_size Process in chunks
warn_memory_threshold_mb Memory warning threshold

validate_delta_table()

Validate Delta table by name

table_name e.g., "bronze.incidents"
batch_name Optional batch identifier

validate_lakehouse_file()

Validate files in Lakehouse

file_path e.g., "Files/data/file.parquet"
file_format parquet, csv, json, delta

handle_failure()

Handle validation failures

results Results from validate_*
action "log", "halt", or "alert"

Example Configurations

Ready-to-use validation templates for common scenarios

🥉 Bronze Layer: Raw Data Ingestion

Basic structure and schema validation. Log failures, don't halt.

bronze_validation.yml 
validation_name: bronze_incidents
description: Raw incident data validation

quality_thresholds:
  critical: 100
  high: 95
  medium: 80
  low: 50

expectations:
  # Table-level checks
  - expectation_type: expect_table_row_count_to_be_between
    kwargs:
      min_value: 1
      max_value: null
    meta:
      severity: critical
      description: Table should not be empty

  # Required columns exist
  - expectation_type: expect_column_to_exist
    kwargs:
      column: incident_id
    meta:
      severity: critical
      
  - expectation_type: expect_column_to_exist
    kwargs:
      column: created_date
    meta:
      severity: critical

  # Basic type validation
  - expectation_type: expect_column_values_to_not_be_null
    kwargs:
      column: incident_id
    meta:
      severity: high
      description: Primary key cannot be null

🥈 Silver Layer: Cleaned Data

Business rules and data quality checks. Raise on critical failures.

silver_validation.yml 
validation_name: silver_incidents
description: Cleaned incident data validation

quality_thresholds:
  critical: 100
  high: 98
  medium: 90
  low: 70

expectations:
  # Uniqueness
  - expectation_type: expect_column_values_to_be_unique
    kwargs:
      column: incident_id
    meta:
      severity: critical
      description: Primary key must be unique

  # Foreign key integrity
  - expectation_type: expect_column_values_to_not_be_null
    kwargs:
      column: category_id
    meta:
      severity: high

  # Valid category values
  - expectation_type: expect_column_values_to_be_in_set
    kwargs:
      column: status
      value_set: ["Open", "In Progress", "Resolved", "Closed"]
    meta:
      severity: medium

  # Date range validation
  - expectation_type: expect_column_values_to_be_between
    kwargs:
      column: priority
      min_value: 1
      max_value: 5
    meta:
      severity: medium

🥇 Gold Layer: Business-Ready

Strict validation for analytics. Always halt on any failure.

gold_validation.yml 
validation_name: gold_incident_metrics
description: Business metrics validation

quality_thresholds:
  critical: 100
  high: 100
  medium: 100
  low: 95

expectations:
  # Zero tolerance for nulls in dimensions
  - expectation_type: expect_column_values_to_not_be_null
    kwargs:
      column: incident_id
    meta:
      severity: critical

  - expectation_type: expect_column_values_to_not_be_null
    kwargs:
      column: category_name
    meta:
      severity: critical

  # Metric integrity
  - expectation_type: expect_column_values_to_be_between
    kwargs:
      column: resolution_hours
      min_value: 0
      max_value: 8760  # Max 1 year
    meta:
      severity: high
      description: Resolution time must be positive and reasonable

  # Aggregation accuracy
  - expectation_type: expect_column_mean_to_be_between
    kwargs:
      column: resolution_hours
      min_value: 0.5
      max_value: 168  # 1 week average max
    meta:
      severity: medium
      description: Average resolution should be under 1 week
